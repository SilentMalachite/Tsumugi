# Claude Code 実装指示 — Tsumugi フェーズ1（マスタ ＋ 記録）

> 役割: 実装。親文書 `01_ClaudeCode_実装指示書_Tsumugi.md` のフェーズ1を詳細化したもの。
> 着手条件: **フェーズ0の受け入れ基準（AC0-1〜AC0-6）が再レビューで適合済み**であること。
> レビューは `02_Codex_レビュー指示書_Tsumugi.md` の観点で実施する。

---

## 1. 結論
事業所・利用者・受給者証・契約・日次記録（サービス提供実績＋出欠）のCRUDを、**データ性質に応じた3つの永続化方針**で実装する。受給者証の**期限アラート**と日次記録の**訂正・取消履歴**を純粋ロジックとして固定し、追記不変・楽観ロックをテストで強制する。報酬算定に必要な値（単位数・加算コード）は**ここでは確定せず**、構造だけ用意してフェーズ3へ渡す。

---

## 2. 前提

### 2.1 永続化の3方針（フェーズ1の中核判断）
| 区分 | 対象 | 方針 |
|---|---|---|
| 取引記録 | 日次記録（実績＋出欠） | **厳密追記**。更新・削除しない。訂正・取消は新レコードで表現し履歴を残す。 |
| 期間マスタ | 受給者証・契約・事業所体制 | **実効日付つき追記**。期間 `[開始日, 終了日]` を持ち、「基準日時点」を期間で引く。訂正は新行で上書きしない。 |
| 同一性マスタ | 事業所・利用者 | **楽観ロック（更新トークン）＋監査**（作成/更新の who/when）。可変だが破壊更新の痕跡を残す。 |

### 2.2 日付の取り扱い（既知の地雷。Ayumiの境界バグの教訓を持ち込む）
- すべての期間は **両端含む `[開始日, 終了日]`**、**暦日（ローカル日付、時刻成分なし）**で比較する。UTC変換に起因する日跨ぎを発生させない。
- 「基準日 D 時点で有効」= `開始日 ≤ D ≤ 終了日`。
- **日付検証はApplication層（信頼境界）で行う**。UIの入力制限だけに依存しない。
- `DateOnly` を用い、`DateTime`/タイムゾーンを記録ロジックに混入させない。

### 2.3 報酬依存値の非確定（§ハード制約3の再掲）
- サービスコード・加算の**単位数や具体コードはフェーズ1で確定しない**。事業所体制の加算フラグは「**拡張可能なコード集合**」として構造のみ用意し、実値・正式コードは `docs/open-questions.md` に残してフェーズ3（報酬告示参照）で確定。

## 3. スコープ

### 3.1 含む
- エンティティ6種のCRUD（§4）、受給者証期限アラート、日次記録の訂正・取消、対応するAvalonia画面（MVVM）。
- 追記不変の**強制機構**（SaveChangesガード）と楽観ロックの実装・テスト。

### 3.2 含まない
- 報酬算定・工賃計算・CSV/帳票出力（フェーズ2・3）。
- サービスコード解決・加算単位数（フェーズ3）。
- 個別支援計画の立案。

### 3.3 親文書からの明示的なリファイン（意図的変更）
- **出欠は独立集約にせず、日次記録の属性（状態）として持つ**（出席/欠席/欠席時対応/中止 等）。エンティティ数を減らし状態追跡を簡素化。`01 §4.3` の「出欠」を本書で日次記録に統合する。
- 日次記録は**提供実態（事実）のみ**を記録し、報酬サービスコードへの解決はフェーズ3に置く（`01 §4.3` の「サービスコード」を後ろ倒し）。

## 4. 仕様

### 4.1 エンティティ（最小要件。拡張可。すべて `Guid` 主キー、`作成日時`/`作成者` を持つ）

**Office（事業所・同一性マスタ）**
- 事業所番号、名称、サービス種別（B型を初期値）、地域区分。
- 更新トークン（`Guid`）＋更新監査。

**OfficeCapability（事業所体制・期間マスタ／実効日付つき追記）**
- `OfficeId`、適用期間 `[開始日, 終了日?]`（終了日nullで継続中）。
- **加算体制フラグ群＝拡張可能なコード集合**（`体制加算コード → 有効/無効`）。コードの正式値・単位数はフェーズ3で確定するため、ここでは「コード（文字列/列挙）＋有効フラグ」の保持に留める。代表例は実装コメントに *「要・報酬告示突合（暫定）」* と明記し、確定済みのように扱わない。

**Recipient（利用者・同一性マスタ）**
- 氏名（漢字・カナ）、生年月日。
- 更新トークン＋更新監査。

**Certificate（受給者証・期間マスタ／実効日付つき追記）**
- `RecipientId`、受給者証番号、有効期間 `[開始日, 終了日]`、支給量（日数）、利用者負担上限月額、市町村。
- 同一利用者で**期間の重複・空白を検出**して警告（ハードブロックはしない。ポリシーはADRに記す）。

**Contract（契約・期間マスタ／実効日付つき追記）**
- `RecipientId`、利用開始日、利用終了日（null可）、契約支給量。

**DailyRecord（日次記録＝サービス提供実績＋出欠・取引記録／厳密追記）**
- `RecipientId`、`ServiceDate`（`DateOnly`）。
- 区分 `RecordKind`：新規 / 訂正 / 取消。
- `訂正元Id`（`Guid?`、訂正・取消時に対象を指す）。
- 出欠状態（出席/欠席/欠席時対応/中止 等の列挙）。
- 提供実態：送迎（なし/往/復/往復）、食事提供の有無、提供の有無、提供時間または提供形態 など**事実のみ**。
- **更新トークンは持たない**（更新しないため）。

### 4.2 訂正・取消の表現（純粋ロジック）
- レコードは決して更新・削除しない。訂正は `区分=訂正`＋`訂正元Id` の新レコード、取消は `区分=取消`＋`訂正元Id` の新レコードを追加する。
- **実効状態の導出**は純粋関数：`(あるRecipientとServiceDateの全レコード) → 実効レコード | なし`。最新の有効な訂正を辿り、取消されていれば「なし」。
- 月次ビュー＝`(Recipient, 期間)` に対する実効レコード集合（純粋関数）。フェーズ2/3はこの実効集合を入力にする。

### 4.3 受給者証 期限アラート（純粋ロジック）
- `アラート対象(certificates, 基準日, しきい値日数) → [(Recipient, Certificate, 残日数)]`。
- `残日数 = 終了日 − 基準日`（暦日）。対象条件：`0 ≤ 残日数 ≤ しきい値`。`残日数 < 0` は失効。
- 純粋関数。日付/I/Oに依存しない。

### 4.4 追記不変の強制（テストで歯を持たせる）
- `DbContext` の `SaveChanges`/`SaveChangesAsync` を**インターセプタまたはオーバーライド**で監視し、**取引記録エンティティ（DailyRecord）が `Modified`/`Deleted` 状態のときは例外で落とす**。期間マスタも同様に「破壊更新」を禁止（新行のみ許可）。
- これにより「追記型」を規約でなく**実行時不変条件**にする。

### 4.5 楽観的同時実行
- 同一性マスタ（Office/Recipient）は更新トークン（`Guid`）を保存時に更新し、競合時に検知して例外を返す。SQLiteに `rowversion` がないため**更新トークン列方式**。

### 4.6 画面（Avalonia / MVVM。ViewModelをUIなしで単体テスト可能に）
- 利用者一覧・編集、受給者証管理（**期限アラート一覧を含む**）、契約、事業所・体制、日次記録入力（月グリッドで実績＋出欠、訂正・取消操作）。
- **頻用操作はキーボードで完結**。ダークテーマ・低アニメーションが既定（`01 §6-5`/アクセシビリティ）。
- 入力検証メッセージは利用者を責めない平易な文言。日付・支給量・番号書式を検証。

## 5. TDD

### 5.1 Domain（純粋ロジック中心、テーブル駆動）
- 期限アラート：境界網羅（残日数＝しきい値ちょうど／0／負＝失効／受給者証なし／複数受給者証）。
- 訂正・取消：新規のみ／訂正1回／多段訂正／取消後の再訂正／取消で「なし」になる、を固定。
- 期間判定：両端含む、月初・月末、`開始日＝終了日`、期間重複・空白検出。

### 5.2 Application（モックリポジトリ）
- 日付検証が信頼境界で効くこと（UIを通さない不正日付を弾く）。
- 受給者証期間重複の警告がユースケースで surfacing される。

### 5.3 Infrastructure（一時ファイルSQLite）
- 6エンティティのCRUD往復、マイグレーション適用。
- **追記不変ガード**：DailyRecordをUpdate/Deleteしようとすると例外（意図的違反テストで赤を確認）。
- 楽観ロック：同一行の競合更新で確実に検知。
- 期間マスタの「基準日時点」クエリが正しい行を返す。

### 5.4 App（ViewModel）
- 期限アラートViewModelがしきい値で対象を出す。
- 日次記録ViewModelの訂正・取消コマンドが追記として反映され、実効状態が更新される。

## 6. 制約 / 品質ゲート（`01 §6` に加えフェーズ1固有）
- `01 §6` の全項目を維持（オフライン＝App含む、DB権限、nullable/警告ゼロ、依存方向、format、クロスプラットフォーム、バックアップ）。
- **追記不変**が実行時に強制され、違反テストで赤になる。
- 日付ロジックは `DateOnly` ベースで、タイムゾーン依存の比較を持たない。
- 報酬依存値（単位数・正式加算コード）のハードコードなし。暫定値は *要・報酬告示突合* と明記し open-questions に起票。
- 新規の設計判断（出欠の統合、期間重複ポリシー、加算フラグの保持形）は `docs/decisions/` にADR化。

## 7. 受け入れ基準（チェック可能）
- [ ] (AC1-1) 事業所・事業所体制・利用者・受給者証・契約・日次記録をCRUDできる。
- [ ] (AC1-2) DailyRecordが**厳密追記**で、Update/Delete企図が実行時例外で落ちる（違反テストで赤を確認済み）。
- [ ] (AC1-3) 日次記録の訂正・取消が元を破壊せず、実効状態の導出（純粋関数）が多段訂正・取消を正しく扱う。
- [ ] (AC1-4) 期間マスタ（受給者証・契約・事業所体制）が実効日付つき追記で、「基準日時点」クエリが正しい行を返す。
- [ ] (AC1-5) 受給者証期限アラートが、しきい値日数で対象を抽出（境界網羅のテーブル駆動テスト）。失効・複数証・証なしを正しく扱う。
- [ ] (AC1-6) 同一性マスタの楽観ロックが競合を検知。
- [ ] (AC1-7) 日付検証がApplication層（信頼境界）で効き、`DateOnly`・両端含む・暦日比較で固定。
- [ ] (AC1-8) 主要操作がキーボードで完結し、ダークテーマ・低アニメーションが既定。期限アラート一覧が画面で確認できる。
- [ ] (AC1-9) 加算フラグ構造が拡張可能なコード集合として用意され、実値は未確定として open-questions に残る（フェーズ1で報酬値を確定していない）。

## 8. 着手手順
1. Domainに6エンティティ（record）と純粋ロジック（期限アラート・実効状態導出・期間判定）を、まずテストから定義。
2. Infrastructureにマイグレーションと追記不変ガード・楽観ロックを実装し、一時SQLiteで往復テスト。
3. Applicationにユースケースと信頼境界の日付検証を追加。
4. AppにViewModelと画面を追加（ViewModelテスト先行）。
5. `./build/ci.sh` 緑・`dotnet format` 通過・依存方向不変を確認してフェーズ1受け入れへ。

> 公式仕様から一意に決まらない点（加算の正式コード体系、受給者証期間重複の業務ルール等）は推測で埋めず `docs/open-questions.md` に起票し、設計相談へ回す。
