# Claude Code 実装指示書 — Tsumugi（ポチパス相当 / .NET 10 + Avalonia 新規プロジェクト）

> 役割: **実装**。本書に従って新規プロジェクトを起こし、フェーズ順にTDDで実装する。
> コードネーム `Tsumugi`（紡ぎ）は仮称。`src/` 配下のプレフィックスごと自由に改名可。
> レビューは別書（`02_Codex_レビュー指示書`）でCodexが担当する。本書はレビュー指摘を取り込む受け皿でもある。

---

## 1. 結論

就労継続支援B型事業所向けに、ポチパス相当の機能（**記録 ＋ 工賃 ＋ 国保連請求データ生成**）を持つ **オフライン専用デスクトップアプリ** を、.NET 10 / Avalonia / EF Core(SQLite) で新規に構築する。フルセットを4フェーズに分割し、各フェーズの受け入れ基準を満たしてから次へ進む。報酬算定値と請求CSV仕様は**コードに直書きせず外部化**し、報酬改定で差し替え可能にする。

---

## 2. 前提

### 2.1 環境
- .NET SDK **10.0**（`net10.0` ターゲット、LTS）
- Avalonia **11.x**（Fluent テーマ）
- EF Core **10.x** ＋ `Microsoft.EntityFrameworkCore.Sqlite`
- IDE: Rider（非商用ライセンス）。CLIビルド（`dotnet build` / `dotnet test`）が常に通ること。
- OS: 開発は macOS、運用は施設内Windows端末を想定 → **クロスプラットフォームで動くこと**を常に確認。
- ネットワーク: **完全オフライン**。外部通信・クラウド依存・テレメトリは禁止。

### 2.2 ドメイン
- サービス種別: **就労継続支援B型** を第一対象（他種別は将来拡張、設計上は閉じない）。
- 個人情報（氏名・受給者証番号・障害情報）を扱う前提でデータ保護を設計する。

### 2.3 重要な責務境界（誤解防止）
- 国保連請求は **「請求データ(CSV)を仕様準拠で生成し、公式の取込・送信システムにインポートできる状態まで」** が責務。
- **電子請求受付システムへの伝送・電子証明書・回線処理は範囲外**（アプリは行わない）。
- 報酬の **単位数・加算・地域区分単価・CSVフィールド定義** は本書では数値を確定しない。実装者は**公式の報酬告示および「障害者総合支援　電子請求　インターフェース仕様書」から取得**し、外部マスタ（DBテーブル＋シードJSON）として投入する。出典が確認できない数値はハードコードしない。

### 2.4 認知特性に基づく設計方針（実装者向けの恒久指針）
- 可変状態の追跡コストを最小化する。エンティティは可能な限り **C# `record` ＋ 追記型（append-only）** で表現し、訂正は新レコードで履歴を残す（Ayumi踏襲）。
- 報酬算定・工賃計算は **副作用のない純粋関数**として実装し、入出力を値で閉じる。
- 1画面1責務。状態が画面をまたいで暗黙に共有される構造を作らない。

## 3. スコープ

### 3.1 含む（フルセット）
1. マスタ・記録（フェーズ1）
2. 工賃計算（フェーズ2）
3. 国保連請求データ生成（フェーズ3）

### 3.2 含まない
- 国保連への伝送・送信・電子証明書管理
- 個別支援計画の作成支援（記録参照は可、計画立案ロジックは対象外）
- 給与（職員）計算、会計・仕訳
- マルチテナント／クラウド同期

### 3.3 フェーズ分割（この順に実装。前フェーズの受け入れ基準充足が次の着手条件）
- **フェーズ0**: プロジェクト基盤（ソリューション構成・DI・EF Core・テスト基盤・CIスクリプト・バックアップ）
- **フェーズ1**: マスタ＋記録（事業所／利用者／受給者証／契約／サービス提供実績／出欠）
- **フェーズ2**: 工賃計算（月次集計・PDF/印刷）
- **フェーズ3**: 国保連請求データ生成（報酬算定エンジン・実績記録票・請求明細書・CSV出力）

## 4. 仕様

### 4.1 ソリューション構成（依存方向を厳守）
```
Tsumugi.sln
  src/
    Tsumugi.Domain/          エンティティ(record)・値オブジェクト・報酬算定/工賃の純粋ロジック。外部依存なし。
    Tsumugi.Application/     ユースケース・リポジトリ抽象(interface)・DTO・入力検証。Domainのみ参照。
    Tsumugi.Infrastructure/  EF Core DbContext・SQLite・リポジトリ実装・バックアップ・CSV出力。Application/Domain参照。
    Tsumugi.App/             Avalonia UI(MVVM)。合成ルート(DI構成)はここ。Application参照＋Infrastructureを起動時にのみ束ねる。
  tests/
    Tsumugi.Domain.Tests/
    Tsumugi.Application.Tests/
    Tsumugi.Infrastructure.Tests/
```
- 依存方向: `App → Application → Domain`、`Infrastructure → Application/Domain`。**DomainはInfrastructure/UIを知らない**。
- UIから`DbContext`を直接触らない。必ずApplication層のユースケース経由。

### 4.2 採用ライブラリ
- MVVM: `CommunityToolkit.Mvvm`（`ObservableObject` / `[RelayCommand]` のソースジェネレータ）
- DI/ホスト: `Microsoft.Extensions.DependencyInjection` ＋ `Microsoft.Extensions.Hosting`
- テスト: `xUnit` ＋ `FluentAssertions`
- PDF/印刷: まずAvalonia標準の印刷経路（A4実績記録票）を最優先。PDF固定出力が必要なら`QuestPDF`（Community/MITライセンス条件を確認の上）を候補とし、採用時はレビューで明示。

### 4.3 データモデル（フェーズ1中心。最小要件、拡張可）
- `事業所(Office)`: 事業所番号・名称・サービス種別・地域区分・体制届情報（加算可否のフラグ群）。
- `利用者(Recipient)`: 氏名（漢字/カナ）・生年月日・障害支援区分等。
- `受給者証(Certificate)`: 受給者証番号・有効期間（開始/終了）・支給量・**期限アラート**（Ayumiの監視ロジックと同等の考え方を新規実装）。
- `契約(Contract)`: 利用開始日・終了日・契約支給量。
- `サービス提供実績(ServiceProvisionRecord)`: 日付・サービスコード・提供形態・送迎有無・食事提供有無等。**追記型**、訂正は訂正レコードで表現。
- `出欠(Attendance)`: 日付・状態（出席/欠席/欠席時対応加算対象 等）。
- 共通: 主キーは`Guid`。`作成日時`/`作成者`を持つ。更新は**楽観的同時実行**（更新トークン列を保存時に更新し、競合検知）。SQLiteに`rowversion`はないため**更新トークン(Guid)方式**で実装する。

### 4.4 報酬算定エンジン（フェーズ3の心臓部）
- マスタ: `単価マスタ(UnitPrice)` を `(サービスコード, 適用開始年月, 地域区分)` で引く。`加算マスタ(Addon)` を体制・実績条件で適用。
- すべての単位数・単価・加算率は **DBテーブル＋シードJSONで外部化**し、`適用開始年月`でバージョン管理。**令和6年改定を初期データとし、令和9年改定を無改修で差し替え可能**にする。
- 算定本体は純粋関数: `算定(実績集合, 事業所体制, マスタ) → 請求明細`。日付・乱数・I/Oに依存しない。
- 端数処理・上限処理・日割り・月跨ぎは仕様として明文化し、テストで固定する（4.7参照）。

### 4.5 国保連請求データ生成（フェーズ3）
- 出力物:
  1. **サービス提供実績記録票**（A4印刷ビュー）
  2. **介護給付費・訓練等給付費等 請求書／明細書**（画面確認＋印刷）
  3. **請求データCSV**（公式インターフェース仕様準拠。取込・送信システムにインポート可能な形式）
- CSVは仕様書のレコード種別・項目順・桁・文字コード・改行コードに厳密準拠。**仕様の出典バージョンをファイルヘッダコメントまたは添付メタに残す**。
- 生成のみ。送信・伝送は行わない。

### 4.6 アクセシビリティ（運用者＝主たる利用者の感覚特性に対する合理的配慮。必須要件）
- 既定で **ダークテーマ**、背景は落ち着いた暗色（例: `#214d4a`）を選べる。
- **アニメーション最小**（Reduce Motion相当）。画面遷移・トランジションは無効化可能、既定で控えめ。
- 余白・行間広め、フォントサイズ拡大に追従。コントラスト確保。
- 長時間入力を想定し、**頻用操作はキーボードで完結**（マウス追従の眼球運動負荷を下げる）。
- これらはテーマ/設定として実装し、ハードコードしない。

### 4.7 計算仕様の固定（曖昧さの排除）
- 工賃: 月次集計の基準（出来高/時間/固定）、締め日、端数処理（円未満切り捨て等）を**設定として明示**し既定値を定める。会計年度（4月開始）対応。
- 報酬: 端数・上限・日割りの規則を仕様コメントとテスト名で1対1対応させる。
- 「仕様が公式資料からは一意に確定できない」箇所は**実装を止め、TODOと根拠リンク欄を残してレビューへ回す**（推測で埋めない）。

## 5. TDD

### 5.1 基本サイクル
- Red → Green → Refactor を厳守。各機能はテストファースト。
- 1コミット＝1論理変更。コミットメッセージにフェーズ番号と受け入れ基準IDを記す。

### 5.2 レイヤ別テスト戦略
- **Domain**: 純粋ロジックの単体テスト中心。報酬算定・工賃・期限アラートは**テーブル駆動テスト**で既知ケースを網羅。
- **Application**: ユースケースをモックリポジトリで検証（入力検証・分岐・例外）。
- **Infrastructure**: 一時ファイルSQLite（テストごとに使い捨てDB）で結合テスト。マイグレーション適用→CRUD→楽観ロック競合→バックアップ(`VACUUM INTO`)の往復を確認。
- **App(ViewModel)**: MVVMによりViewModelをUIなしで単体テスト（コマンド・状態遷移・検証表示）。

### 5.3 重点カバレッジ
- 報酬算定モジュールは**分岐網羅100%を目標**（請求金額に直結＝誤りが事故）。既知の入力→期待明細の対応表をテストとして資産化。
- CSV出力は**バイト単位のスナップショットテスト**（文字コード・改行・桁）。

## 6. 制約 / 品質ゲート

すべてのフェーズ完了の前提条件（CIで機械判定できるものは機械判定）:

1. `dotnet build` 警告ゼロ。`<Nullable>enable</Nullable>` ＋ `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>`。
2. `.editorconfig` ＋ Roslynアナライザ有効。`dotnet format --verify-no-changes` 通過。
3. `dotnet test` 全緑。Domain/Applicationのカバレッジ閾値を設定（報酬算定は100%目標）。
4. 依存方向違反ゼロ（Domainが外側を参照していないことをアーキテクチャテストまたは参照構成で保証）。
5. **オフライン**: 実行時にネットワークを使うコードを持たない。外部通信ライブラリを参照しない。
6. **データ保護**: SQLiteファイルの保存先・権限を明示。個人情報を含むため、ログに氏名・受給者証番号を出さない。暗号化（SQLCipher等）採用可否はフェーズ0で判断しレビューに明記。
7. **バックアップ**: `VACUUM INTO` による単一ファイルバックアップ手段を提供（Ayumi踏襲）。
8. **報酬告示・CSV仕様の外部化**: 単位数/加算/CSVフィールドのハードコード禁止。出典バージョンをデータに保持。
9. アクセシビリティ要件（4.6）が既定で有効。
10. クロスプラットフォーム: macOS/Windows双方で起動・主要操作が動くこと（最低限、CLIビルドとViewModelテストは両OSで緑）。

> 上記の各文書化済み判断（暗号化採否、PDFライブラリ採否、端数規則など）は、コミット内`docs/decisions/`にADR（1件1ファイル、結論→背景→選択肢→決定→影響）として残す。

## 7. 受け入れ基準（フェーズ別・チェック可能）

### フェーズ0（基盤）
- [ ] ソリューションが §4.1 の構成・依存方向で生成され、`dotnet build`/`dotnet test` が緑。
- [ ] DI合成ルートが`App`にあり、`Infrastructure`差し替えがテストで可能。
- [ ] EF Core初期マイグレーションが適用でき、一時ファイルSQLiteで往復テストが通る。
- [ ] `VACUUM INTO` バックアップのユースケースとテストが存在。
- [ ] 品質ゲート §6 の 1〜5 がCIで判定される。

### フェーズ1（マスタ＋記録）
- [ ] 事業所/利用者/受給者証/契約/サービス提供実績/出欠 をCRUDでき、追記型・楽観ロックが効く。
- [ ] 受給者証期限アラートが、しきい値日数で対象を抽出できる（テーブル駆動テストあり）。
- [ ] サービス提供実績の訂正が、元データを破壊せず訂正履歴として残る。
- [ ] 主要操作がキーボードで完結し、ダークテーマ・低アニメーションが既定。

### フェーズ2（工賃）
- [ ] 月次工賃が、設定した基準・締め・端数規則どおりに集計される（既知ケースのテーブル駆動テスト）。
- [ ] 会計年度（4月開始）での月次集計とPDF/印刷出力ができる。

### フェーズ3（国保連請求データ生成）
- [ ] 報酬算定が、外部マスタ（令和6年改定シード）に基づき請求明細を生成（分岐網羅100%目標、既知ケース表で検証）。
- [ ] サービス提供実績記録票のA4印刷ビューが出力できる。
- [ ] 請求書/明細書を画面確認・印刷できる。
- [ ] 請求データCSVが、公式インターフェース仕様（出典バージョンを明記）に準拠し、バイト単位スナップショットテストが緑。
- [ ] マスタの`適用開始年月`差し替えで、算定結果が無改修で切り替わることをテストで実証。

## 8. 着手手順（最初の1コミット）
1. §4.1 のソリューション骨格を生成（空のクラスライブラリ＋テストプロジェクト、参照関係を設定）。
2. 依存方向を壊さないことを最小のアーキテクチャテストで固定。
3. `dotnet build`/`dotnet test` 緑、`dotnet format` 通過を確認してコミット（メッセージ: `phase0: solution skeleton`）。
4. 以降はフェーズ0の残受け入れ基準をテストファーストで埋める。

> 不明点・公式仕様から一意に確定できない点は、推測で実装せず `docs/open-questions.md` に列挙し、Codexレビューおよび設計相談へ回すこと。
