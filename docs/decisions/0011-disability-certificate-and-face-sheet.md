# ADR 0011: 障害者手帳とフェースシートを利用者と 1:N の追記専用エンティティとして実装する

- 結論: 障害者手帳 (`DisabilityCertificate`) とフェースシート (`FaceSheet`) を、利用者 (`Recipient`) と 1:N の関係を持つ追記専用 (`AppendOnlyGuard` 対象) エンティティとして実装する。等級改定・更新・記載内容変更は **新レコードを追記** し、最新の「現行版」を取り出す UseCase を別に提供する。等級表記やフェースシート各項目は様式差を考慮し可能な限り文字列で保持する。
- 背景:
  - 受給者証 (ADR 0010) は MHLW 公式様式で項目が一意に定まる一方、**障害者手帳**は身体／療育／精神の 3 法律根拠で等級体系も自治体運用も異なる。例:
    - 身体: 1〜6級、種別「1種/2種」
    - 療育: 「A1/A2/B1/B2」「1〜4度」「重度/中軽度」など自治体表記
    - 精神: 1〜3級、2 年ごとの更新が必要
  - **フェースシート**は事業所ごとに項目が完全に標準化されておらず、自由記述の比率が高い。一方、連絡先・主治医・緊急連絡先・受給状況等の主要グループは現場運用で必ず必要になる。
  - これらを利用者編集画面に詰め込むと画面が破綻するうえ、CLAUDE.md「1 画面 1 責務」原則と整合しない。
- 選択肢:
  - (a) **採用**: 別エンティティ＋追記。利用者ごとに履歴を保つ。UI は新規タブで recipient selector ＋一覧＋追加フォーム / 編集フォームを提供。
  - (b) Recipient エンティティに列を直接追加。1:1 のみ。シンプルだが履歴を持てず、Recipient マスタが肥大化する。フェースシートの更新も「上書き」になり履歴破壊。
  - (c) JSON 1 列に格納。型安全性・索引・将来の請求/介護保険連携突合性が損なわれる。
- 決定: (a)。理由:
  - 障害者手帳は等級が変動する制度設計（精神は 2 年ごと更新等）。追記により再認定の履歴が残る。
  - フェースシートも「いつ誰が更新したか」が支援記録上重要。append-only + 最新版取得が自然。
  - CLAUDE.md ハード制約 (record + 追記型) と整合。`AppendOnlyGuard` に両型を登録し、UPDATE/DELETE は機械的に拒否。
  - UI は「利用者選択 → タブ内で履歴/編集」のパターンを既存の受給者証/契約 View と統一できる。
- 影響:
  - DB マイグレーション `20260627223018_AddDisabilityCertificateAndFaceSheet` を生成（実値は手元で再生成日時に依存）。
  - 新タブ 2 つを追加 (`障害者手帳(_3)` / `フェースシート(_4)`)。受給者証以降のタブは `(_5)〜(_9)` に繰り下げ。
  - `FaceSheet` は履歴が残るため、ストレージ容量は更新頻度に比例して増える。`VACUUM INTO` バックアップ運用には影響なし。Phase 3 で履歴圧縮・古バージョン要約等の検討余地あり。
  - 等級表記・自治体差・フェースシートの自治体/事業所固有項目は `docs/open-questions.md` に追記して Phase 2/3 で対応する。
