# ADR 0022: 利用者負担上限額マスタと月次上限額管理結果の優先関係

**ステータス**: 確定（2026-07-10）

## 結論

`PaymentBurdenCategory` の制度上限額、受給者証に記載された `Certificate.MonthlyCostCap`、利用者負担上限額管理結果票の事業所別「管理結果後利用者負担額」を別の入力として保持する。請求時は制度マスタを検証用の補助上限として使い、受給者証の個別実値を上書きしない。上限額管理対象者は、成人B型に適用できる一次資料が登録された期間だけ、正式な管理結果票の該当事業所行を最終利用者負担額とする。現行の資料集合でこの経路を使えるのは2026-06以降のR8だけであり、R6成人B型は上限額管理対象の場合だけ算定不能とする。対象外の通常上限制御は継続する。

区分・受給者証実値・対象判定・必要な管理結果票のいずれかが未入力、未知、期間外または不整合なら、0円に補正せず算定を停止する。0円は、出典と入力済み状態を確認できる公式マスタ値、受給者証記載値または正式な管理結果後金額だけを許可する。

## 背景

既存の `PaymentBurdenCategory` は `Unspecified`、`Welfare`、`LowIncome`、`General1`、`General2` を持ち、`Certificate` は必須の非負 `int MonthlyCostCap` と、nullableな上限額管理事業所名だけを持つ。この形では「未入力の既定値0」と「受給者証に記載された0円」を区別できず、上限額管理の対象有無や管理結果票も表現できない。

制度上限額は所得区分の一般的な上限である。一方、請求に用いる個別上限は市町村が認定して受給者証へ記載する値であり、複数事業所を利用する対象者の最終負担額は月次の管理結果票で事業所別に調整される。制度マスタだけから個別値または調整結果を生成すると、市町村判断と正式帳票を推測することになる。

## 選択肢

### A: 3入力を分離し、制度値を検証用の補助上限にする（採用）

- 制度区分、受給者証の個別認定、月次の事業所間調整を監査可能にできる。
- 未入力0と正式な0円を区別できる。
- 既存モデルに入力済み状態と管理結果票を追加する必要がある。

### B: `PaymentBurdenCategory` だけから請求上限を決める

- 実装は単純だが、受給者証の個別実値と市町村判断を上書きするため却下する。

### C: `Certificate.MonthlyCostCap` だけを使い、上限額管理を事業所内で再計算する

- 受給者証実値は使えるが、正式な管理結果票による事業所別調整を無視し、複数事業所の負担を誤るため却下する。

## 決定

### 制度上限額マスタ

マスタIDを `burden-cap-master-r6-04` とし、サービス提供月で有効な行を選ぶ。`effectiveFrom` / `effectiveTo` は両端を含むサービス月である。現時点で終了月が公示されていないため `effectiveTo` はnullとするが、将来改定時に既存行を上書きせず、新しい期間行を追加する。

| key | displayName | effectiveFrom | effectiveTo | amountYen | sourceDocumentId | 物理頁 | 適用条件 |
| --- | --- | --- | --- | ---: | --- | ---: | --- |
| `welfare` | 生活保護 | 2024-04 | null | 0 | `r6-disability-support-guide-202404` | 9 | 生活保護受給世帯 |
| `low-income` | 低所得 | 2024-04 | null | 0 | `r6-disability-support-guide-202404` | 9 | 市町村民税非課税世帯 |
| `general-1` | 一般1 | 2024-04 | null | 9,300 | `r6-disability-support-guide-202404` | 9 | 市町村民税課税世帯のうち所得割16万円未満。ただし20歳以上の入所施設利用者とグループホーム利用者を除く |
| `general-2` | 一般2 | 2024-04 | null | 37,200 | `r6-disability-support-guide-202404` | 9 | 上記以外。20歳以上の入所施設利用者とグループホーム利用者は、市町村民税課税世帯なら一般2 |

`PaymentBurdenCategory` との対応は `Welfare -> welfare`、`LowIncome -> low-income`、`General1 -> general-1`、`General2 -> general-2` の完全一致だけを許可する。`Unspecified` は制度額0の区分ではなく非入力状態なので、マスタ行を作らず算定不能とする。未知のenum値または文字列を近い区分へ寄せない。

制度額の根拠は、ADR 0020の全3版束に登録した `r6-disability-support-guide-202404` 物理9頁である。R8の `r8-grant-decision-administration-202606` 物理112頁は制度額を別資料参照としており、金額自体の根拠には使わない。

### 3入力の責務境界

| 入力 | 責務 | してはならないこと |
| --- | --- | --- |
| 制度上限額マスタ | サービス月と `PaymentBurdenCategory` に対応する一般的な上限額を返し、受給者証実値と管理結果の上界を検証する | 受給者証実値を生成・上書きする、市町村の区分認定を推測する |
| `Certificate.MonthlyCostCap` | 市町村が認定し、サービス日に有効な受給者証へ記載した個別の負担上限月額を保持する | 制度マスタの値で自動補完する、未入力0を記載0と扱う |
| 月次の上限額管理結果 | 正式な利用者負担上限額管理結果票にある各事業所の管理結果後利用者負担額を保持する | 事業所体制、制度区分または暫定上限だけから事業所間配分を推測する |

R8では市町村が所得区分等を認定して受給者証に負担上限月額と適用期間を記載することを、`r8-grant-decision-administration-202606` 物理112〜113頁で確認する。R6の通常の上限制御は `r6-disability-support-guide-202404` 物理9頁の制度値と、サービス日に有効な受給者証の登録スナップショットを使う。`r6-claim-handbook-202405` は2024-05だけに有効であり、18〜19頁の結果票規則は障害児通所用なので、R6成人B型の結果票検証にも2024-04への遡及にも使わない。

### 請求時の優先手順

次の順序を変更せず、途中の検証に失敗したら後続へ進まない。

1. 受給者と各サービス提供日に対して有効な `Certificate` をちょうど1件選ぶ。月内に証が切り替わり、請求単位の区分・実値・管理対象情報を一意にできない場合は推測せず拒否する。
2. 選択した証の `Certificate.PaymentBurden` を区分の唯一の権威ソースとし、同じ証から `MonthlyCostCap`、上限額管理対象状態、上限額管理事業所を同時に読む。別の請求入力に区分を複製せず、`Unspecified`、未知値または入力済み状態を確認できない値を拒否する。
3. 証の `PaymentBurden` とサービス提供月から、有効な制度上限額マスタ版と対応行をそれぞれちょうど1件選ぶ。版なし、期間重複、行なし、重複key、負額を拒否し、`masterCap` を検証用の補助上限として得る。
4. 証の区分とマスタkeyの対応、マスタ行の適用期間・適用条件、`certificateCap` が入力済みの非負値で `certificateCap <= masterCap` であることを検証する。低い個別実値は許可するが、制度マスタから区分または証実値を補完しない。
5. 各事業所の上限適用前利用者負担額 `rawBurden` を、単位数・単価・給付率等の別契約により算出する。`provisionalBurden = min(rawBurden, masterCap, certificateCap)` とし、これを当該事業所の上限額管理前の暫定値とする。
6. 受給者証上、上限額管理対象外であることが明示されている場合は、管理結果票が存在しないことを確認し、`provisionalBurden` を最終利用者負担額とする。対象外なのに管理結果票または管理事業所がある場合は不整合として拒否する。
7. 上限額管理対象である場合は、成人B型に適用できる一次資料の版をサービス月で解決する。2024-04〜2026-05はR6成人用資料が未登録なので、結果票が入力されていても算定不能とする。2026-06以降だけ `r8-grant-decision-administration-202606` を選び、R8規則をR6へ遡及しない。
8. 手順7でR8版を解決できた場合に限り、同じサービス月・受給者・受給者証・上限額管理事業所に対応する確定済みの正式な管理結果票をちょうど1件選ぶ。請求する事業所番号の行、証上限スナップショット、管理結果区分、各事業所の上限管理前利用者負担額、管理結果後利用者負担額、各合計を検証する。全行で `0 <= 管理結果後額 <= 上限管理前額`、管理結果後合計が `certificateCap` と `masterCap` 以下、当該事業所の上限管理前額が手順5の暫定値と一致することを要求する。
9. 検証済み管理結果票の請求対象事業所行にある「管理結果後利用者負担額」を最終利用者負担額として採用する。暫定値、制度額または受給者証実値で正式結果を置換しない。

次の管理結果検証は2026-06以降だけ、`r8-grant-decision-administration-202606` 物理173〜186頁を根拠に適用する。管理結果区分は公式の `1`（管理事業所で充当し他事業所負担なし）、`2`（合算額が証上限以下で調整なし）、`3`（合算額が証上限を超え調整）だけを許可する。区分1は管理事業所の管理結果後額と管理結果後合計が証上限に一致し、管理事業所以外の管理結果後額が0であることを要求する。区分2は全行で管理結果後額と上限管理前額が一致し、両合計も一致することを要求する。区分3は上限管理前合計が証上限を超え、管理結果後合計が証上限と一致することを要求する。

`r6-claim-handbook-202405` 物理18〜19頁にも同名の結果区分と金額関係があるが、明示的に「障害児通所の場合」の規則である。同資料の有効日は2024-05であり、成人B型にも2024-04にも適用しない。R6成人用の結果票規則を示す一次資料が不変バイトとSHA-256を伴ってADR 0020へ登録されるまで、R6成人の上限額管理対象は結果票検証と事業所配分を行わず算定不能とする。上限額管理対象外なら手順6で通常の暫定上限を最終値にでき、この制限の対象外である。

### 上限額管理の対象判定

対象有無は次の三値を構造化入力する。

| 値 | 意味 | 必須の整合条件 |
| --- | --- | --- |
| `Applicable` | 上限額管理対象 | 受給者証に上限額管理事業所が記載され、事業所番号を一意に解決でき、成人B型に適用できる当月版の一次資料と正式結果票がある |
| `NotApplicable` | 上限額管理対象外 | 受給者証に対象外と明示され、管理事業所と結果票がない |
| `Unknown` | 未確認・未入力 | 算定不能 |

複数サービスの利用や上限超過の可能性だけから対象有無を推測しない。既存の `Certificate.UpperLimitManagementProvider` は名称しか持たず、nullも「対象外」と「未入力」を区別できないため、単独では請求根拠にしない。対象者が上限額管理者を届け出て、市町村が受給者証へ管理事業所を記載する一般フローは、2024-05の境界確認に限り `r6-claim-handbook-202405` 物理16頁、2026-06以降は `r8-grant-decision-administration-202606` 物理176〜179頁を根拠とする。

### 正式な管理結果票の必要最小入力

Phase 3-1で次を構造化入力として追加する。既存Domainには相当する月次実体がないため、本ADRではフィールド契約だけを確定し、文字列メモで代用しない。入力構造が存在しても、適用できる成人用一次資料がないR6では算定に使用しない。

| フィールド | 必須内容 |
| --- | --- |
| 識別 | サービス提供月、支給決定市町村番号、受給者証番号、受給者ID |
| 証スナップショット | 利用者負担上限月額、上限額管理対象状態、受給者証に記載された上限額管理事業所番号 |
| 管理事業所 | 指定事業所番号、名称 |
| 帳票状態 | 原本に記載された作成区分、受領日時、原本参照ID、確定状態 |
| 管理結果 | 管理結果区分 `1` / `2` / `3` |
| 事業所行 | 項番、事業所番号、名称、総費用額、上限額管理前利用者負担額、管理結果後利用者負担額 |
| 合計 | 総費用額合計、上限額管理前利用者負担額合計、管理結果後利用者負担額合計 |

作成区分は原本の表示値をそのまま保持し、障害児通所用資料にあるコード意味を成人B型へ流用しない。どの原本が確定版かを一意に解決できない場合は未確定として拒否する。R8の帳票単位、管理結果、事業所行と合計は `r8-grant-decision-administration-202606` 物理182〜186頁を根拠とする。

### 未入力0と正式な0円

Phase 3-1では、少なくとも `Certificate.MonthlyCostCap` と管理結果票の金額を「入力有無」と金額に分ける。永続化・DTO・UIの表現はnullableな金額と明示的な入力状態を持ち、`IsEntered = false` と `ValueYen = 0` を同一にしない。既存の必須 `int MonthlyCostCap` だけではこの区別を証明できないため、移行前の値0を一律に正式値へ昇格しない。受給者証原本を再確認し、入力済み状態と原本参照を登録してから請求に使う。

0円を合法とするのは、次のいずれかで必要な出典・入力・期間がすべて揃った場合だけである。

- `Welfare` / `LowIncome` に対応する公式制度マスタ値0円。
- 有効な受給者証に明記され、入力済み状態と原本参照を持つ `MonthlyCostCap = 0`。
- 適用できる成人用一次資料により検証済みの正式な管理結果票にある、当該事業所の管理結果後利用者負担額0円。現行資料では2026-06以降のR8だけを許可する。
- 別契約で正しく算出された `rawBurden = 0`。

未指定区分、証未登録、管理結果票未着またはパース失敗を0円へ倒してはならない。

### フェイルクローズ条件

次の条件はすべて算定不能とし、0円の請求結果を生成しない。

| 境界 | 拒否条件 |
| --- | --- |
| 制度区分 | `Unspecified`、未知値、対応keyなし、適用条件を確認できない |
| マスタ | サービス月の版なし、行なし、期間重複、key重複、負額、未知の `sourceDocumentId` |
| 受給者証 | サービス日に有効な証なし、複数証で一意でない、`Certificate.PaymentBurden` とマスタkeyの対応不一致、原本未確認、登録スナップショットと原本の不一致 |
| 証上限 | 未入力、入力済み状態なし、負額、制度上限超過、サービス月・有効期間との不整合 |
| 対象判定 | `Unknown`、対象状態と管理事業所の欠落・存在が矛盾、受給者証原本と登録スナップショットが不一致 |
| 管理結果票 | 対象なのに適用できる成人用一次資料がない、結果票が存在しない、未確定、重複、月・受給者・証・管理事業所・請求事業所の不一致 |
| 管理結果額 | 負額、上限管理前額超過、証上限または制度上限超過、行合計不一致、結果区分1〜3との関係不一致 |
| 対象外 | 管理事業所または管理結果票が存在する、対象外であることを受給者証から確認できない |

国保連合会の受給者台帳との突合は、現行Domainにも上記の構造化入力契約にも存在しないため、本ADRの必須検証に含めない。本ADRで必須とする境界は受給者証原本と登録スナップショットの整合までである。国保連合会の外部審査はこの結果を別途拒否でき、将来台帳を構造化入力する場合は別契約として追加するが、現時点のAC3-0/Phase 3-1実装へ存在しない入力を要求しない。エラー時に「安全側」と称して0円を採用することは禁止する。

### 成人B型の事業所間徴収順

本ADRが確定するのは、制度マスタ、受給者証実値、正式な管理結果票のデータ優先関係であり、管理結果票を新規生成するための事業所間配分規則ではない。

`r6-claim-handbook-202405` 物理18〜19頁の徴収順、管理結果区分と金額関係は「障害児通所の場合」であり、成人の就労継続支援B型へ適用しない。R6成人B型の結果票規則と徴収優先順は現在の一次資料集合では確定できないため、Phase 3-1で成人の結果票検証と複数事業所配分を実装しない。外部の結果票が入力されても、成人用一次資料を追加するまでフェイルクローズする。

R8の成人を含む順序は `r8-grant-decision-administration-202606` 物理183〜185頁に、上限額管理事業所、日中活動系、訪問系、就労定着支援・自立生活援助、短期入所、共同生活援助（体験利用）の順として示される。この順序を使えるのは2026-06以降の適用版だけであり、R6へ遡及しない。R8で配分機能を追加する場合も、同資料に基づく別実装・別テストで扱い、本ADRの優先手順から暗黙生成しない。

| sourceDocumentId | 本ADRで使用する物理頁 | 用途 |
| --- | --- | --- |
| `r6-disability-support-guide-202404` | 9 | 4区分の条件と0円 / 0円 / 9,300円 / 37,200円 |
| `r6-claim-handbook-202405` | 16〜20 | 2024-05の一般フローと適用境界の確認に限定する。18〜19頁の障害児通所用の徴収順、結果1〜3、金額関係、作成区分は成人B型へ使用せず、2024-04にも遡及しない |

## 影響

- AC3-0では、この契約に従う版付き制度上限額マスタと厳格な検索を実装できる。
- Phase 3-1では、受給者証上限の入力済み状態・原本参照、三値の上限額管理対象状態、管理事業所番号、月次管理結果票実体と行を追加する必要がある。
- 既存の `MonthlyCostCap = 0` と `UpperLimitManagementProvider = null` は移行時に意味を推測せず、原本確認待ちとして扱う。
- R6成人B型の上限額管理対象は、結果票の有無にかかわらず成人用一次資料を登録するまで算定不能となる。対象外の通常上限制御は継続できる。
- 制度区分マスタの改定時は有効期間を追加し、過去行と受給者証実値を上書きしない。
