# Codex レビュー指示書 — Tsumugi（ポチパス相当 / .NET 10 + Avalonia 新規プロジェクト）

> 役割: **レビュー / 監査**。実装は別書（`01_ClaudeCode_実装指示書`）でClaude Codeが行う。
> 本書は実装を変更するためでなく、**受け入れ基準と品質ゲートへの適合を検証し、修正可能な指摘を出力する**ためのもの。
> 指摘は最終的にClaude Code向け指示書へトリアージして戻す（従来のレビュー運用を踏襲）。

---

## 1. 結論

実装書（`01`）に定義された4フェーズの成果物を、**受け入れ基準・品質ゲート・ドメイン正確性・データ保護**の観点でレビューする。特に **報酬算定の正しさ**、**請求CSVの仕様準拠**、**レイヤ依存方向**、**個人情報の扱い** を最重点とする。各指摘は重大度付き・再現手順付き・修正方針付きで出力する。

---

## 2. 前提

- レビュー対象は `Tsumugi.sln`（`src/` ＋ `tests/`）と `docs/`（ADR・open-questions）。
- 受け入れ基準・品質ゲートの正本は `01_ClaudeCode_実装指示書` の §6・§7。本書はそれに対する**適合判定の手順**を定める。
- 責務境界の確認: 国保連請求は **CSV生成まで** が正。**伝送・電子証明書処理が実装に紛れ込んでいたら指摘**（スコープ逸脱）。
- 報酬単位数・加算・CSVフィールドは外部マスタ化されている前提。**ハードコードされていたら指摘**。
- レビュアは数値の正誤を独自に断定しない。公式報酬告示・インターフェース仕様書と**実装が突合可能な形（出典バージョン記録）になっているか**を見る。出典欠落は指摘対象。

---

## 3. スコープ

### 3.1 レビューする
- アーキテクチャ境界・依存方向
- 報酬算定／工賃計算の正確性とテスト網羅
- EF Core 利用の妥当性（マイグレーション整合・トランザクション境界・N+1・楽観ロック）
- 請求CSV仕様準拠（文字コード・改行・桁・項目順）
- データ保護（個人情報のログ漏れ・保存先・暗号化判断）
- アクセシビリティ既定値（ダーク・低アニメ・キーボード操作）
- クロスプラットフォーム整合
- テスト品質（意味のある表明か、カバレッジの質）

### 3.2 レビューしない
- 機能追加・設計の作り直し提案（必要なら別途オープン質問として起票）
- スコープ外項目（伝送・職員給与・会計）の作り込み——存在したら「逸脱」として指摘するのみ

## 4. 仕様（レビュー観点チェックリスト）

### 4.1 アーキテクチャ／依存方向
- [ ] `Domain` が `Infrastructure`/`App`/Avalonia/EF Core を参照していない。
- [ ] UIから`DbContext`直接操作がない（Application経由）。
- [ ] 合成ルートが`App`に集約され、`Infrastructure`が差し替え可能。
- [ ] `record`＋追記型の方針が崩れ、可変状態の暗黙共有が生じていないか。

### 4.2 報酬算定（最重点）
- [ ] 算定が純粋関数で、日付/乱数/I/Oに依存しない。
- [ ] 単位数・加算・地域区分単価が外部マスタ＋`適用開始年月`でバージョン管理され、ハードコードがない。
- [ ] 端数・上限・日割り・月跨ぎの規則がテスト名と1対1で固定されている。
- [ ] 既知ケースの入力→期待明細の対応表テストが存在し、**分岐網羅が実質的に達成**されている（カバレッジ数値だけでなく、境界・例外ケースの有無を見る）。
- [ ] `適用開始年月`差し替えで結果が無改修で切り替わるテストがある。

### 4.3 工賃計算
- [ ] 締め・端数・基準（出来高/時間/固定）が設定化され、既定値が明示。
- [ ] 会計年度（4月開始）の月次集計が正しい。テーブル駆動テストの網羅性を確認。

### 4.4 EF Core / 永続化
- [ ] マイグレーションが一貫し、空DBから現行スキーマまで適用できる。
- [ ] トランザクション境界がユースケース単位で適切。部分コミットの危険がない。
- [ ] 一覧取得にN+1がない（必要な`Include`/射影）。
- [ ] 楽観ロック（更新トークン方式）が競合を確実に検知し、テストがある。
- [ ] `VACUUM INTO` バックアップが機能し、復元往復が検証されている。

### 4.5 請求CSV / 帳票
- [ ] CSVの文字コード・改行コード・レコード種別・項目順・桁が仕様準拠。
- [ ] バイト単位スナップショットテストがあり、差分が説明可能。
- [ ] 出典（インターフェース仕様書のバージョン）がデータ/ヘッダに記録されている。
- [ ] 実績記録票A4・請求書/明細書が出力でき、レイアウト崩れがない。

### 4.6 データ保護 / セキュリティ
- [ ] ログ・例外メッセージに氏名・受給者証番号など個人情報が出ていない。
- [ ] SQLite保存先・ファイル権限が明示され、暗号化採否がADRで判断済み。
- [ ] 外部通信コード・テレメトリ・クラウド依存が**一切ない**（オフライン厳守）。

### 4.7 アクセシビリティ / UX
- [ ] 既定でダークテーマ・低アニメーション・広めの余白／行間。
- [ ] 頻用操作がキーボードで完結。
- [ ] フォント拡大に追従し、コントラストが確保されている。

### 4.8 クロスプラットフォーム
- [ ] macOS/Windows双方でCLIビルドとViewModelテストが緑。
- [ ] パス区切り・改行・カルチャ依存（数値/日付整形）にOS差の地雷がない。

## 5. レビュー手順

1. クリーン環境で `dotnet build`（警告ゼロ確認）→ `dotnet test`（全緑・カバレッジ確認）→ `dotnet format --verify-no-changes` を実行し、品質ゲート §6(01) の機械判定項目を再現。
2. 対象フェーズの受け入れ基準（§7 of `01`）を1項目ずつ、対応するテスト/コードに突合。
3. §4 の観点チェックリストを上から走査。指摘を §6 の重大度で分類。
4. `docs/open-questions.md` と `docs/decisions/`（ADR）を確認し、未解決の仕様確定事項・推測実装の有無を洗い出す。
5. スコープ逸脱（伝送実装・職員給与・会計の混入）が無いか確認。

## 6. 制約 / 品質ゲート（レビュー出力の規約）

### 6.1 重大度
- **Blocker**: 請求金額誤り・個人情報漏れ・依存方向崩壊・オフライン違反・スコープ逸脱。フェーズ受け入れ不可。
- **Major**: テスト不足（境界/例外欠落）、N+1、トランザクション境界の欠陥、CSV仕様の軽微な不適合。
- **Minor**: 命名・整形・ドキュメント不足・改善余地。

### 6.2 指摘フォーマット（1件1ブロック、そのままClaude Code指示書に転記できる形）
```
[重大度] 表題
- 場所: ファイル:行 / テスト名 / 受け入れ基準ID
- 事象: 何が基準・仕様に反するか（事実ベース、推測と区別）
- 再現: 手順またはテスト
- 期待: あるべき挙動／基準
- 修正方針: Claude Codeが着手できる粒度の指示（実装は書かない、方針を示す）
- 根拠: 受け入れ基準ID / 仕様出典（不明なら「要確認」と明記）
```

### 6.3 出力の原則
- 数値の正否を独自確定しない。出典突合可能性で判断し、未確定は「要確認」として起票。
- 1指摘＝1論点。まとめない。
- 「良い実装」も簡潔に記録（リグレッション防止の意図）。
